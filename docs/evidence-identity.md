# Evidence identity and structural grounding

Open Review Atlas keeps source identifiers readable without treating them as globally unique.
This distinction is essential when many repositories use names such as `claim-1` or `ref-1`.

## Local and global identifiers

- `localClaimId` and `localCitationId` are the identifiers asserted by one source repository.
- `claimId` and `citationId` are Atlas identifiers scoped by the immutable `ReviewVersion.id`.
- Global ids use `oratlas:{claim|citation}:v1:{encoded-version}:{encoded-local-id}`.
- Each component is encoded as four lowercase hexadecimal digits per JavaScript UTF-16 code unit.
  This is total and injective for every JavaScript string, including an unpaired surrogate; no
  variable-width concatenation or delimiter ambiguity can create a collision.
- DOM claim anchors are generated by Atlas from the same version/local-id pair. A repository's
  anchor is retained as `sourceAnchor` metadata but never controls a platform DOM id.

These identifiers are durable application identifiers, not content hashes. The immutable version
is separately bound to its exact 40-character SHA-1 or 64-character SHA-256 Git object id.

## Scholarly-work aliases

The same work can be cited with different repository-local ids. Atlas compares canonical aliases:

- `doi:10.…` — resolver/prefix removed and normalized to lowercase;
- `pmid:…` — prefix/URL and insignificant leading zeroes removed;
- `openalex:W…` — URL/prefix removed and the work id uppercased.

Cross-review shared-citation proposals intersect these aliases, never local citation ids. Atlas
does not invent a crosswalk when no common alias exists. If a connected alias cluster asserts two
different values for the same scheme (for example, one PMID connected to two DOIs), the packet and
review API expose an explicit conflict assertion and automated linking does not merge that cluster.

## Evidence packet 1.1

Each packet carries the selected versions, exact snapshot Git object ids, globally scoped and local
evidence ids, canonical work aliases, claim→citation relations, TRUST provenance, and identifier
conflicts. LLM output uses explicit per-statement `evidenceEdges`:

```json
{
  "statement": "…",
  "evidenceEdges": [{ "claimId": "…", "citationId": "…" }]
}
```

Every edge must exist in the packet. The union of the per-statement edge identifiers must exactly
equal `reviewClaimsUsed` and `citationsUsed`. This blocks a citation attached to claim A from being
laundered as evidence for claim B merely because both ids occur in the same packet.

For LLM mode, Atlas validates and canonicalizes the packet once. The exact canonical JSON byte
string is SHA-256 hashed, sent unchanged as the provider message, and stored unchanged on the
`AgentRun`; prompt version `atlas-discuss-2.0` records the output contract.

This validation establishes **structural grounding and provenance only**. It does not establish
that a claim, source, TRUST assessment, or generated answer is scientifically correct.

## Durable version routes

- UI: `/reviews/{slug}/versions/{versionId}`
- API: `/api/reviews/{slug}/versions/{versionId}`
- Read-only comments API: `/api/reviews/{slug}/versions/{versionId}/comments`

Each route verifies that the version belongs to the slug and reads the selected version's own
snapshot, repository, metadata, claims, citations, identifiers, and version-bound discussion.
Historical pages never post, reply to, or remove comments.
